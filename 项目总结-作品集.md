# 马年朋友圈祝福贺卡生成器 · 项目总结（AI 产品经理实习作品集）

---

## 一、项目概述 (Project Overview)

**产品名称**：朋友圈画像智能新年祝福语生成 Agent（马年祝福贺卡生成器）

**核心定位**：基于朋友圈截图的多模态分析，智能生成个性化新年祝福语，并输出风格化贺卡，解决祝福语同质化、撰写耗时、与对象性格/关系不匹配等问题。

**目标用户**：需在春节等场景向亲友、同事、师长发送祝福的普通用户；偏好个性化、二次元/文艺/职场等细分场景的年轻与职场用户。

**我的角色**：独立完成从 PRD 撰写、需求拆解、到前后端落地与联调的全流程，并完成多轮迭代（异常兜底、移动端与分享、去水印、体验优化等）。

**技术栈**：前端 React + Vite + TypeScript + Tailwind + shadcn/ui；后端 FastAPI + Provider 抽象（LLM / Vision / T2I / Storage）；多模态与文生图接入火山引擎豆包（Doubao）与 Seedream。

---

## 二、产品背景与问题定义 (Problem & Opportunity)

- **痛点**：传统新年祝福语同质化严重，难以根据祝福对象的性格与社交形象定制；用户手动撰写耗时且贴合度低。
- **机会**：结合多模态图像理解、大模型文本生成与文生图能力，用「朋友圈截图 + 关系选择」作为输入，输出「性格画像 → 多风格祝福语 → 定制贺卡」的端到端体验。
- **价值**：提升祝福的个性化与仪式感，降低用户构思与排版成本，并可作为 AI 能力落地的完整案例（多模态、Prompt 设计、关系与风格匹配、异常兜底）。

## 三、功能模块与实现范围 (Scope & Implementation)

### 3.1 截图上传与预处理 (PRD 4.1)

- **输入规则**：支持 JPG/PNG/JPEG，单张 ≤10MB；要求截图包含朋友圈个人主页完整界面。
- **实现**：前端单图上传、格式与大小约束；上传前提示「四区域」要求（头像、背景、个性签名、动态区）。后端接收文件后写入存储（OSS 或本地），返回可访问 URL 供后续视觉与展示使用。
- **存储策略**：优先阿里云 OSS；若 OSS 请求失败（如 SSL/网络异常），自动回退到本地 `static` 目录，保证流程不中断。

### 3.2 多模态信息提取与解析 (PRD 4.2)

- **需求**：按四区域（头像、背景、个性签名、动态）做结构化提取，输出 5 条客观信息，供性格画像与贺卡风格使用。
- **实现**：使用豆包多模态模型（doubao-seed-1.8），按 PRD 4.2.3 的 Prompt 严格输出：① 头像描述 ② 朋友圈背景图描述 ③ 个性签名原文 ④ 朋友圈动态内容 ⑤ 朋友圈可见范围。后端对模型回复做解析（按条目拆分），得到结构化 `extraction` 对象并贯穿后续链路。
- **异常**：若返回「未收到有效的朋友圈截图」或调用异常，标记为识别失败，走通用祝福语与贺卡兜底。

### 3.3 用户性格画像生成 (PRD 4.3)

- **需求**：基于 5 条提取信息生成标准化标签（风格/性格/年龄倾向）与一段综合性格画像；头像风格 100% 主导，背景仅作补充。
- **实现**：LLM 调用采用 PRD 4.3.3 的 Prompt，要求「逐元素解读 + 综合性格画像」，语气温柔、有画面感、避免空洞形容词。有截图时使用 `analyze_personality_from_extraction`；无截图时仅基于关系与用户勾选的性格标签生成简短画像。LLM 超时或网络错误时自动回退到本地规则拼接的画像，保证主流程可完成。

### 3.4 用户关系与祝福逻辑 (PRD 4.4)

- **需求**：按祝福对象关系（亲属/朋友/同事/师生及二级关系）约束语气、用词与禁忌，并与 2026 丙午马年示例对齐。
- **实现**：关系选项配置于 `data/relations.json`（一级：亲戚/朋友/同事/师生；二级：长辈/平辈/晚辈、密友/普通朋友、领导/同级/下级、老师/学生）。祝福语与贺卡 Prompt 的生成中，将关系标签与 PRD 4.4 的「关系-风格匹配」规则写入 System/User Prompt，由大模型按关系生成得体内容。

### 3.5 多风格祝福语生成 (PRD 4.5)

- **需求**：一次生成三种风格（文言风、文艺风、大白话），每条 20–50 字；用户选择一条后再进入贺卡生成；支持重新生成与历史回溯。
- **实现**：单次 LLM 调用按 PRD 4.5.3 的 Prompt 输出三行（文言风：… / 文艺风：… / 大白话：…），后端解析后返回 `blessingOptions` 数组。前端展示三条可点击选择，提供「重新生成三条」与「历史第 N 组」下拉；重新生成调用 `POST /api/regenerate-blessings`，不重新解析图片。LLM 超时或失败时回退到本地三条固定马年祝福语。

### 3.6 智能贺卡生成与导出 (PRD 4.6)

- **需求**：未选贺卡风格时按头像风格自动匹配；已选则按用户选择；贺卡需包含祝福语与马年元素；支持祝福语大小与称呼替换（如「您」→「舅妈」）；不生成真人脸。
- **实现**：
  - **两阶段接口**：`POST /api/generate` 只做到「三类祝福语 + 性格/关系/提取信息」；用户选一条祝福语后，前端再调 `POST /api/generate-card`，传入选中祝福语、风格、关系、性格、extraction 等，后端生成贺卡图并返回 URL。
  - **风格匹配**：`style_key=auto` 时根据头像/背景描述与性格画像推断风格（二次元→赛博、佛系/中老年→国风、文艺/简约→手写等）；用户选择具体风格则直接采用。
  - **贺卡 Prompt**：按 PRD 4.6.2 由 LLM 生成英文绘图指令（竖版、马年元素、祝福语写在贺卡上、风格贴合关系与性格）；若检测到真人/自拍类头像，在 Prompt 中强制加入「不生成真人面部、仅插画/剪影/装饰」类约束。
  - **手动调整**：前端提供贺卡风格下拉、祝福语大小（小/中/大）、称呼替换两个输入框，替换后的文案再传给 `/api/generate-card`。
  - **文生图**：请求豆包 Seedream 文生图 API，设置 `watermark: false` 以去除右下角「AI生成」水印；生成后可选保存至 OSS 或本地并返回最终 URL。贺卡保存失败时同样回退到本地存储。

### 3.7 辅助功能与异常兜底 (PRD 4.7)

- **重新上传**：用户再次选择截图并提交时，覆盖当次解析与祝福语结果。
- **祝福语历史**：本次会话内每次「重新生成三条」追加一组到历史列表，用户可通过「第 N 组」下拉回溯并从中选择一条。
- **异常兜底**：视觉提取失败或返回无效截图 → 使用通用马年三类祝福语并标记 `useFallback`；LLM 超时/网络错误 → 性格画像、三条祝福、贺卡 Prompt 分别回退到本地规则或模板；存储（OSS）失败 → 上传与贺卡保存回退到本地 `static`。保证在各种异常下仍能完成端到端流程或给出明确提示。

## 四、主流程与接口设计 (Flow & API)

**主流程顺序（PRD 5.1）**：  
上传截图 → 预处理/存储 → 五条结构化提取 → 性格画像展示 → 生成三类祝福语 → 用户选择一条（可重新生成或从历史选）→ 按头像/用户选择确定贺卡风格 → 可选手动调整祝福语与称呼 → 生成贺卡 → 下载/分享。

**核心接口**：

- `POST /api/generate`：表单（关系、性格标签、可选截图）；返回 `blessingOptions`、`relation`、`personalityProfile`、`extraction`、`imageUrl`、`providers`、`useFallback`。
- `POST /api/regenerate-blessings`：Body 含 relation、personality_profile、personality_tags；返回新的 `blessingOptions`。
- `POST /api/generate-card`：Body 含 `selected_blessing_text`、`style_key`、`relation`、`personality_profile`、`extraction` 等；返回 `cardImageUrl`。
- 原子能力接口：`/api/upload-image`、`/api/parse-relation`、`/api/analyze-image`、`/api/generate-blessing`、`/api/generate-prompt` 等，供拆分调用或后续扩展。

## 五、技术架构与可扩展性 (Architecture)

- **后端**：FastAPI 应用；Provider 抽象（LLM / Vision / T2I / Storage），每种能力可有 Mock 与真实实现（如豆包 LLM/Vision、Seedream T2I、阿里云 OSS），通过环境变量切换，业务层无感知。
- **前端**：单页应用，两栏布局（表单 + 结果/贺卡预览），响应式与移动端 viewport；API 基地址通过 `VITE_API_BASE` 配置，便于部署到公网并分享。
- **可扩展点**：新增关系类型、祝福风格、贺卡风格可在配置与 Prompt 中扩展；更换 LLM/Vision/T2I/Storage 只需实现对应 Provider 并配置环境变量。

---

## 六、产品与体验决策 (Product Decisions)

- **两阶段生成**：先出三条祝福语、用户选一条再出贺卡，避免一次性生成过多内容且提升「选择感」与可控性。
- **头像主导 + 真人脸约束**：风格以头像为准并写入 Prompt；若识别为真人/自拍，强制在贺卡生成中禁止真人面部，兼顾个性化与合规/观感。
- **多层兜底**：存储、视觉、LLM、贺卡保存均具备回退逻辑，在 API 不稳定或网络异常时仍可完成流程或给出通用结果，减少「白屏/报错」体验。
- **移动端与分享**：Viewport、响应式布局与触控友好控件；部署后同一链接可在手机浏览器使用，并支持将链接分享给他人使用。
